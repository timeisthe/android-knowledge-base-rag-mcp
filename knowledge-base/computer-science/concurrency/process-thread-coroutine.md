# 进程、线程与协程的关系

## 概述

程序是存放在磁盘上的静态代码和数据；进程是程序运行后形成的资源容器和隔离边界；线程是进程内部真正执行指令、参与 CPU 调度的执行流；协程则是由语言和运行时管理的可挂起任务，它需要依托线程执行，但自身不是操作系统线程。

可以用一条关系链概括：

`程序 → 进程 → 线程 → 协程任务`

这条关系表示承载关系，不表示每层严格一一对应。一个应用可能使用多个进程，一个进程通常包含多个线程，一个线程也可以在不同时刻执行大量协程。

## 进程：资源容器与隔离边界

操作系统加载并运行程序时，会创建进程并为其建立相对独立的运行环境。通常所说“进程是资源分配的基本单位”，重点不是所有资源都只能按进程分配，而是进程构成了主要的资源所有权和故障隔离边界。

一个进程通常关联：

- 独立的虚拟地址空间；
- 程序代码、全局数据和堆；
- 文件描述符或系统句柄；
- 安全身份、权限和环境变量；
- IPC、信号及其他内核资源；
- 一个或多个线程。

不同进程默认不能直接访问彼此的地址空间，需要通过 Binder、Socket、共享内存、管道等 IPC 机制交换数据。这种隔离提高了安全性和稳定性，但进程创建、切换和通信的成本通常也高于同一进程内的线程协作。

“一个应用等于一个进程”只是常见情况，不是严格规则。例如 Android 应用可以通过组件的 `android:process` 配置运行多个进程，系统也可能杀死应用进程，之后再创建新进程恢复组件。

## 线程：CPU 调度的执行流

进程本身提供资源环境，线程负责在这个环境中实际执行指令。现代操作系统通常以可运行的内核线程作为 CPU 调度对象，因此常说“线程是 CPU 调度的基本单位”。

同一进程内的线程共享：

- 代码段；
- 全局变量和堆对象；
- 文件描述符；
- 大部分进程级资源。

每个线程独立拥有：

- 线程 ID；
- 程序计数器；
- CPU 寄存器上下文；
- 调用栈；
- 线程局部存储；
- 调度状态与优先级。

线程的价值不只是因为 CPU 性能增强。它主要解决以下问题：

1. **响应性**：耗时任务执行时，其他线程仍可处理用户输入或系统事件。
2. **并发性**：多个任务在时间上交错推进。
3. **I/O 重叠**：某个线程等待网络或磁盘时，其他线程可以继续工作。
4. **并行性**：多核 CPU 可以让多个线程在同一时刻真正执行。
5. **低成本共享**：线程共享进程资源，协作通常比跨进程通信直接。

## 线程状态与调度

线程从创建到结束通常会经历以下逻辑状态：

`新建 → 可运行 → 运行中 → 阻塞或等待 → 可运行 → 终止`

只有处于可运行状态的线程才能竞争 CPU。线程因为等待锁、I/O、定时器或调度条件而阻塞时，CPU 会执行其他可运行线程。

操作系统保存当前线程的寄存器、程序计数器和栈位置，再恢复另一个线程的上下文，这一过程称为上下文切换。线程并不是数量越多越好；过多线程会增加内存占用、调度开销和缓存失效。

## 共享数据为什么会出问题

同一进程内的线程共享堆和全局数据，因此多个线程访问可变数据时可能产生数据竞争。

数据竞争通常需要同时满足：

- 多个线程访问同一份可变数据；
- 至少一个线程执行写操作；
- 访问之间缺少锁、原子操作或其他明确的 happens-before 关系。

并发问题不只发生在多核 CPU 上。单核 CPU 虽然不能同时执行两个线程，但线程可能在一组操作中间被切换，因此仍可能出现竞态条件。多核环境进一步带来真正的并行执行，以及缓存一致性、内存可见性和指令重排序问题。

常见的控制方式包括：

- 互斥锁和 synchronized；
- 原子变量；
- 不可变数据；
- 线程封闭；
- 消息传递；
- 明确的并发数据结构；
- 遵守语言内存模型建立的 happens-before 关系。

## 协程：可挂起的并发任务

<span id="coroutine-overview" class="kb-anchor-offset"></span>

协程不是线程，也不应简单理解成“对线程的封装”。更准确地说，协程是语言和运行时提供的轻量级并发抽象：它把一个逻辑任务表示成可暂停、保存状态并在之后恢复的状态机。

> 需要继续追踪“为什么从线程/回调演化到协程，以及 `viewModelScope.launch -> SuspendLambda -> delay -> Handler -> resume` 如何完整执行”时，可进入 <a href="/computer-science/concurrency/kotlin-coroutine-suspension-and-resumption#kotlin-coroutine-deep-dive" target="_self">Kotlin 协程挂起、恢复与取消源码深挖</a>。

以 Kotlin 协程为例，挂起函数在编译后会保存：

- 当前执行位置；
- 需要跨挂起点保留的局部变量；
- 恢复执行所需的 Continuation。

当协程遇到真正的挂起操作时，它可以暂时让出当前线程。线程随后可以执行其他协程；异步结果就绪后，调度器再安排该协程继续运行。

因此：

- 一个线程可以交替执行大量协程；
- 一个协程恢复前后可能运行在不同线程；
- 协程可以并发，但不一定并行；
- 是否并行取决于 Dispatcher、线程池和 CPU 核心；
- 挂起不会阻塞线程，但普通阻塞调用仍会占用线程；
- 协程不会自动消除共享可变状态导致的数据竞争。

## “同步写法实现异步调用”的根本原因

协程允许代码按照接近顺序执行的形式编写异步流程：

```kotlin
suspend fun loadScreen(): ScreenData = coroutineScope {
    val user = async { loadUser() }
    val messages = async { loadMessages() }

    ScreenData(
        user = user.await(),
        messages = messages.await()
    )
}
```

代码阅读起来像同步流程，但 `await()` 等挂起点不会要求线程一直空等。编译器生成的状态机会在异步结果到达后，从保存的位置继续执行。

这改善的是异步代码的表达和结构化管理，并不是把所有同步函数自动变成非阻塞函数。如果 `loadUser()` 内部直接执行阻塞 I/O，它仍会阻塞承载该协程的线程，除非使用合适的异步 API或调度到适当的线程池。

## 三者的核心区别

| 维度 | 进程 | 线程 | 协程 |
|---|---|---|---|
| 管理者 | 操作系统内核 | 操作系统内核 | 语言、库和运行时 |
| 核心作用 | 资源管理与隔离 | 执行指令和 CPU 调度 | 组织可挂起的并发任务 |
| 地址空间 | 通常独立 | 共享所属进程地址空间 | 使用承载线程所在进程的内存 |
| 切换成本 | 通常最高 | 次之 | 通常较低 |
| 是否直接使用 CPU | 通过进程内线程 | 是 | 必须在线程上执行 |
| 是否天然并行 | 多进程可并行 | 多线程可并行 | 取决于实际使用的线程 |
| 数据隔离 | 默认较强 | 默认共享 | 默认仍可能共享 |

## Android 中的对应关系

在 Android 中：

- 应用组件运行在应用进程内，但应用不保证永远只有一个进程。
- 每个进程创建后通常有一个主线程。
- 主线程通过 Looper 和 MessageQueue 处理界面绘制、输入事件和生命周期回调。
- UI 对象通常只能在主线程访问。
- `Dispatchers.Main` 将协程安排到主线程。
- `Dispatchers.IO` 适合执行允许阻塞的 I/O 工作。
- `Dispatchers.Default` 适合 CPU 密集型计算。
- `lifecycleScope`、`viewModelScope` 负责把协程任务和 Android 组件生命周期绑定。

使用协程并不意味着可以忽略线程规则。例如协程从 `Dispatchers.IO` 获取数据后，UI 更新仍应回到主线程对应的上下文；多个协程并行修改同一个对象时，仍然需要同步或状态隔离。

## 容易误解的边界

1. **进程不是磁盘上的程序本身。** 程序是静态文件，进程是它的一次运行实例。
2. **出现线程不只是因为 CPU 变强。** 响应性、I/O 等待和共享资源协作同样重要。
3. **并发不等于并行。** 并发是多个任务在一段时间内推进；并行是多个任务同一时刻执行。
4. **单核也会发生竞态。** 指令执行过程中发生线程切换就可能破坏复合操作。
5. **协程不是操作系统调度单位。** 操作系统看到的仍然是线程。
6. **挂起不等于阻塞。** 挂起让出线程；阻塞会占住线程等待。
7. **协程不保证线程安全。** 是否安全取决于共享状态的访问方式。
8. **使用协程不一定切换线程。** 挂起函数可以在同一个线程上暂停并恢复。
