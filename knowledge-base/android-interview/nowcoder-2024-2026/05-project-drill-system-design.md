# 1. 面试官为什么反复拷打项目

项目深挖不是为了听功能介绍，而是在验证四件事：

1. 真实性：你是否真的做过，能否解释代码、数据和故障。
2. 所有权：哪些是你负责，哪些是团队或第三方能力。
3. 技术判断：为什么选这个方案，替代方案是什么。
4. 工程闭环：如何测量、上线、回滚和证明结果。

简历中的“启动优化 30%”“MVI 架构”“跨列表复用”“实时换肤”等词，都会变成追问入口。

# 2. 项目回答的证据链

建议使用以下顺序，而不是只讲 STAR 的故事层：

## 背景与约束

- 用户和业务问题是什么；
- 设备、网络、版本、时间和团队约束；
- 原始指标和故障频率。

## 根因

- 用什么工具和数据定位；
- 排除了哪些假设；
- 最终根因对应哪条线程、引用链、状态流或协议路径。

## 方案

- 数据结构、模块边界、线程模型；
- 为什么选择它；
- 失败或放弃过哪些方案；
- 风险和兼容策略。

## 结果

- 指标采集口径；
- P50/P90/P99 或分机型结果；
- 功能、稳定性和资源成本；
- 灰度、回滚和长期监控。

## 个人贡献

必须具体到设计、编码、排查、推动和评审。不要用“我们做了”掩盖自己的职责。

# 3. 优化类项目的最低合格信息

## 启动优化

必须能回答：

- 冷启动定义；
- 首帧和可交互时间；
- 初始化依赖图；
- 哪些工作移出主线程；
- 异步后如何保证依赖和异常；
- 低端机、弱网、首次安装是否有效；
- 是否增加内存或后续卡顿。

## 内存优化

必须能回答：

- 是泄漏、峰值、抖动还是大对象；
- 用 heap、LeakCanary、Profiler 还是线上指标；
- 关键引用链；
- 修复如何改变所有权；
- 修复后 GC 次数、峰值和 OOM 是否改善。

## RecyclerView 优化

必须能回答：

- 创建和绑定谁更慢；
- item 类型、图片/视频、嵌套列表；
- Diff、payload、稳定 id；
- 复用错位和异步回写；
- 曝光定义；
- 帧时间、卡顿率和内存指标。

# 4. 系统设计题的通用拆法

先向面试官确认：

- 用户规模和数据量；
- 是否需要离线；
- 一致性要求；
- 是否多端同步；
- 低端机和弱网要求；
- 现有技术栈；
- 是否允许服务端改造。

然后按七个层次展开：

1. 数据模型；
2. 状态与单向数据流；
3. 渲染与交互；
4. 线程和异步；
5. 缓存、失败与恢复；
6. 性能、安全和兼容；
7. 日志、指标、灰度和回滚。

# 5. 场景一：即时通讯

## 数据

拆分会话、消息、成员、附件、已读游标和发送任务。为消息建立稳定 id、服务端序号和本地状态。

## 发送流程

UI 创建本地消息 → 写入数据库为 sending → 入发送队列 → 网络发送 → 服务端确认 → 更新 server id/sequence/status → UI 观察数据库变化。

## 接收流程

长连接/推送收到通知 → 去重与顺序校验 → 持久化 → 更新会话摘要和未读 → 通知 UI。

## 边界

- 离线重试和幂等；
- 多端同步；
- 撤回、编辑和引用；
- 大文件上传；
- 数据库分页；
- 推送到达率不等于消息可靠性。

# 6. 场景二：富文本文章与编辑器

## 展示模型

将文章建模为有序 Block 列表，例如 text、image、code、video、quote。每个 Block 有稳定 id、类型、内容和展示元数据。

## 渲染

根据类型映射不同 ViewHolder/Compose 组件；图片和视频异步加载；代码块和长文本需要控制测量成本；大文章使用分页或惰性列表。

## 编辑

用户看到的内容和底层结构不能只靠可见占位符拼接。可选方案包括：

- block editor：每个内容块独立编辑；
- 富文本 span 模型；
- Web 编辑器与原生桥接。

需要处理光标、选择、撤销重做、图片上传中状态、草稿、本地恢复和服务端版本冲突。

# 7. 场景三：复杂首页偶现故障

现象：首帧慢、模块顺序错、旧角标、点击错跳、埋点与真实展示不一致。

## 统一假设

这些问题可能来自同一类竞态：

- 多数据源无版本合并；
- 异步任务完成顺序不稳定；
- 列表复用和 position 过期；
- 状态容器混合持有新旧数据；
- 曝光在 bind 时记录，而非真实可见。

## 排查

- 建立一次启动的 trace id；
- 记录配置、实验桶、数据版本、提交时间和线程；
- 把“请求完成、状态提交、列表绑定、布局、可见”分开打点；
- 检查点击事件是否捕获旧 position；
- 线上采样并按机型、网络、实验桶聚类；
- 用故障注入扩大延迟和乱序，稳定复现。

## 修复

- 统一页面状态和版本；
- 过期响应丢弃；
- item 使用稳定业务 id；
- 只在真实可见且满足时长条件时曝光；
- 首屏依赖设超时与缓存回退；
- 灰度并比较故障率与首帧分位数。

# 8. 场景四：分享模块

目标是屏蔽微信、微博、系统分享等平台差异。

可设计：

- ShareRequest：文本、链接、图片、渠道、追踪参数。
- SharePlatform 接口：validate、share、isInstalled。
- 各平台 Adapter 实现。
- ShareRouter 根据渠道选择实现。
- 统一 Result：success、cancel、error。
- 图片预处理和文件权限模块。
- 埋点与降级策略。

要处理 Activity 生命周期、回调丢失、权限、未安装、超大图片、重复点击和渠道 SDK 线程要求。

# 9. 场景五：朋友圈图片滑动容器

关注点：

- 大图解码和采样；
- 预加载窗口；
- 内存与磁盘缓存；
- 手势冲突；
- ViewPager2/RecyclerView 复用；
- 页面切换时取消过期请求；
- 缩略图到原图的渐进加载；
- OOM、首图时间和滑动帧率。

# 10. 不同轮次如何调整表达

## 一面

先给清晰、正确、覆盖面的答案。不要一开始钻入冷门源码。

## 二面

主动补替代方案、边界和指标。面试官会从数据结构和具体代码验证真实性。

## 三面/社招

从 owner 视角回答：

- 如何拆团队任务；
- 如何推动跨端或服务端改造；
- 风险、灰度和回滚；
- 如何定义成功；
- 出现事故如何止损和复盘。

# 11. 常见低分回答

- “用了 MVVM，所以解耦了”：没有说明状态所有权、数据流和生命周期。
- “把初始化放子线程”：没有依赖、线程安全和首屏结果。
- “用了缓存所以快”：没有 key、过期、一致性和容量。
- “用弱引用解决泄漏”：可能掩盖错误所有权。
- “RecyclerView 有四级缓存”：背名词但说不出每层对象、命中条件和失效。
- “公司框架这样做”：无法证明个人理解与贡献。
