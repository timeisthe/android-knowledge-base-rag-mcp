# 1. 算法考察的真实特点

19 篇合格面经中，14 篇包含算法或现场代码。Android 岗的算法通常不追求竞赛难度，但非常看重：

- 能否快速澄清输入和边界；
- 数据结构选择是否合理；
- 代码能否一次写对；
- 是否主动分析复杂度；
- 能否按要求去掉全局状态、降低空间或改为迭代；
- 代码风格是否适合工程维护。

# 2. 高频题型

## 链表

出现频率最高，包括：

- 反转链表、反转链表 II；
- 环形链表；
- 两数相加；
- 多个有序数组/链表求交；
- 奇偶位置或奇偶序列重排；
- L1-L2-…-Ln 重排为 Ln-L1-Ln-1-L2；
- 不使用额外空间的字符串/链表操作。

训练重点：

- dummy 节点；
- 快慢指针；
- 反转区间；
- 合并；
- 环检测；
- 节点身份与值相等的区别。

## 二叉树与图

包括：

- 层序遍历；
- 最大深度；
- 非递归后序遍历；
- 岛屿最大面积；
- 二维有序矩阵查找。

训练重点：

- BFS 队列；
- DFS 递归与显式栈；
- visited 状态；
- 避免用全局变量返回结果；
- 时间和空间复杂度。

## 哈希、缓存与字符串

包括：

- 大写字母去重；
- LRU；
- HashMap 场景题；
- 字符压缩；
- 回文串；
- 多个有序集合求交。

LRU 应能说明 HashMap + 双向链表的关系，以及 get/put 如何移动到头部、何时淘汰尾部。

## 并发与代码设计

部分公司直接要求写：

- 双重检查锁单例；
- 生产者消费者；
- 死锁示例；
- 子线程创建 Handler；
- 线程超时和串行编排。

这类题同时检查 API 熟练度和对内存模型、生命周期的理解。

## 大数据与外部算法

字节面经出现了“内存远小于文件，求 Top 100 高频词”。合理方向：

1. 按哈希分桶写入多个小文件，使单桶可放入内存；
2. 每桶统计词频；
3. 每桶保留 Top K；
4. 用小顶堆合并得到全局 Top K。

需要讨论倾斜 key、临时磁盘、分桶数量、编码和异常恢复。

# 3. 现场写题流程

1. 复述问题并确认空输入、重复值、范围和是否允许修改原数据。
2. 先给朴素方案，再说明优化依据。
3. 明确不变量，例如链表反转中 prev/current/next 的含义。
4. 编码时保持函数短小，变量名表达角色。
5. 手动走最小样例和边界样例。
6. 给时间、空间复杂度。
7. 主动说明溢出、递归深度、并发和异常边界。

# 4. 面试轮次策略

## 一面

目标：稳定通过基础筛选。

- 回答先给结论和主流程；
- 不确定时明确边界，不要编造源码细节；
- 算法优先保证正确和可读；
- 项目至少准备一个完整优化闭环。

## 二面

目标：展示深度和设计能力。

- 每个方案补替代方案；
- 说明数据结构、线程、生命周期和失败恢复；
- 给量化指标；
- 算法写完接受复杂度或状态管理的二次限制。

## 三面与社招

目标：展示 owner 能力。

- 把问题拆成定位、止损、修复、灰度、验证；
- 说明跨团队依赖；
- 不只讲技术最优，还讲成本和风险；
- 对线上问题给观测与回滚。

# 5. 四周训练计划

## 第 1 周：P0 基础

每天：

- 1 个 Java 集合/并发主题；
- 1 个 Android Framework 主题；
- 2 道链表或树题；
- 口述一次 URL 全流程。

结束标准：

- Handler、Binder、生命周期、HashMap、线程池、TLS 能连续讲 5 分钟；
- 代码题能在 25 分钟内完成并自测。

## 第 2 周：性能和工程

主题：

- 启动优化；
- 内存泄漏；
- RecyclerView；
- OkHttp/缓存；
- Room/IM；
- 包体与动态化。

每天把一个技术点改写成项目证据链。

## 第 3 周：场景设计

至少完成：

- IM；
- 富文本编辑器；
- 复杂首页；
- 分享模块；
- 图片信息流。

每题限时 30 分钟，必须包含数据、状态、线程、生命周期、容错和观测。

## 第 4 周：模拟轮次

- 2 次一面模拟：基础覆盖 + 简历 + 算法。
- 2 次二面模拟：项目深挖 + 系统设计。
- 1 次社招/三面模拟：线上事故和 owner 推进。
- 复盘所有“只会名词、说不出流程”的点。

# 6. 自测清单

如果以下问题不能完整回答，应回到对应主题：

1. MessageQueue 没消息时如何等待，新消息如何唤醒？
2. 为什么 Binder 适合 Android 系统服务，大数据为什么不适合直接传？
3. ViewModel 能跨什么，不能跨什么？
4. HashMap 可变 key 为什么可能取不到值？
5. volatile 为什么不能保证 i++？
6. ConcurrentHashMap 为什么仍不能保证多个操作组合的原子性？
7. LeakCanary 观察对象的时机和最终输出是什么？
8. RecyclerView 异步图片为什么可能错位？
9. HTTPS 中非对称、对称和证书分别解决什么？
10. OkHttp 拦截器、连接池和缓存各在什么位置？
11. 协程挂起后线程去了哪里，状态如何恢复？
12. 启动优化如何证明不是把耗时推迟到首屏之后？
13. 一个 IM 消息从点击发送到对端展示经历哪些状态？
14. 首页模块顺序错、点击错跳和埋点不一致为什么可能是同一根因？
15. LRU 如何保证 get 和 put 都是 O(1)？

# 7. 最后的判断标准

真正准备完成不是“题目看过”，而是：

- 能从定义讲到执行流程；
- 能解释为什么这样设计；
- 能说出错误方案为什么错；
- 能把知识落到项目和指标；
- 能在追问改变约束后调整方案；
- 能写出边界清楚、可运行的代码。
