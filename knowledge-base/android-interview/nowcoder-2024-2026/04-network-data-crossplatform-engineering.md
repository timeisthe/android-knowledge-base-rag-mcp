# 1. 从输入 URL 到页面可用

完整链路可按层次回答：

1. 解析 URL，检查缓存、HSTS、代理等本地策略。
2. DNS 解析域名，可能经过系统缓存、递归解析或 DoH。
3. 建立传输连接：TCP，或 HTTP/3 使用的 QUIC/UDP。
4. HTTPS 场景完成 TLS 握手和证书验证。
5. 发送 HTTP 请求，服务端经过网关、应用和数据层返回响应。
6. 客户端解码、缓存、解析数据并更新 UI。
7. 浏览器场景还涉及 HTML/CSS/JS 解析、布局和绘制；App 场景则进入业务数据绑定和渲染。

面试中应区分“网络协议链路”和“Android UI 渲染链路”，不要混成一个模糊过程。

# 2. TCP、UDP 与可靠性

TCP 的可靠性来自序列号、确认、重传、滑动窗口、流量控制和拥塞控制的组合，不是因为“它有连接”就天然可靠。

三次握手用于双方确认彼此的发送和接收能力，并同步初始序列号。两次无法让服务端确认客户端已收到服务端的序列信息，也更难处理历史连接报文。

下载速度常由慢到快，与拥塞窗口逐步增长有关。现代算法不一定严格沿用经典慢启动行为，但核心仍是先探测路径容量，再控制发送速率。

UDP 不保证到达、顺序和去重，但应用可在其上实现自己的可靠性与低延迟策略。实时音视频常选择 UDP/QUIC，是因为它允许更灵活地处理丢包、时延和部分数据价值。

# 3. HTTPS/TLS

TLS 中：

- 非对称密码和证书体系用于身份认证及安全协商；
- 对称密钥用于后续大批量数据加密；
- 哈希/MAC/AEAD 用于完整性和认证。

证书验证至少包括：

- 域名是否匹配；
- 是否在有效期；
- 证书链能否追溯到受信任根；
- 签名是否有效；
- 撤销和策略检查。

“跳过证书验证”会破坏身份认证，生产环境不能用信任所有证书的方式解决。测试环境也应使用独立 CA、证书 pinning 配置或明确的调试开关。

性能优化方向：

- 连接复用；
- TLS 会话恢复；
- HTTP/2 多路复用；
- 合理的 DNS 和连接预热；
- 减少重定向和握手次数；
- 控制请求体与响应体；
- 避免主线程网络和串行依赖。

# 4. HTTP 版本

## HTTP/1.1

支持长连接和流水线概念，但同一连接上的队头阻塞仍明显，实际客户端通常使用连接池并行请求。

## HTTP/2

把请求拆为帧并在一条连接上多路复用，使用头部压缩。它解决应用层请求串行，但底层 TCP 丢包仍可能让同一连接上的多个流等待。

## HTTP/3

基于 QUIC，在 UDP 上实现可靠传输、加密和流级多路复用，减少连接建立时延，并降低跨流队头阻塞。

# 5. OkHttp 与 Retrofit

## OkHttp

面试时应从调用链回答：

- Request/Call 表示请求；
- Dispatcher 控制同步和异步任务；
- 拦截器链依次处理重试与重定向、桥接、缓存、连接、网络调用等职责；
- ConnectionPool 复用连接；
- Cache 根据 HTTP 缓存语义决定命中、条件请求或网络请求。

应用拦截器更偏业务统一处理；网络拦截器观察的是实际网络层请求和响应。添加统一请求头通常在拦截器完成，但应避免重复、签名顺序错误和缓存键不一致。

文件下载与普通接口不同之处包括：

- 流式写入，避免一次性加载到内存；
- 断点续传和 Range；
- 校验完整性；
- 临时文件与原子替换；
- 进度、取消、网络切换和重试；
- 存储空间与权限。

## Retrofit

Retrofit 的核心是把接口声明解析为请求描述，通过动态代理生成实现，再交给 CallAdapter、Converter 和底层 HTTP 客户端执行。它不是网络传输层本身。

# 6. 缓存与图片加载

缓存设计需要明确：

- 缓存对象和 key；
- 一致性和过期策略；
- 内存、磁盘、HTTP 缓存分别负责什么；
- 淘汰策略与容量上限；
- 多进程和并发写入；
- 弱网回退与陈旧数据标识。

图片库常有活动资源、内存、磁盘资源和原始数据等不同层次，具体层级取决于实现。不能把 HTTP 强缓存、协商缓存和图片库内存缓存简单混为“三级缓存”。

LRU 适合在容量有限时淘汰最久未使用项；实现通常是 HashMap + 双向链表，以 O(1) 完成查找、移动和淘汰。

# 7. Room、IM 与数据模型

## Room

Room 是 SQLite 的抽象层，提供实体映射、DAO、编译期 SQL 校验和迁移机制。

需要能说明：

- 消息、会话、成员、已读状态如何拆表；
- 主键和索引如何选择；
- 事务边界；
- 数据库升级和回滚；
- 分页和历史消息；
- 附件不应直接把大文件存入普通字段，而是保存元数据与文件引用。

## 即时通讯消息

一个可扩展消息模型通常包含：

- messageId、conversationId、senderId；
- serverSequence 或时间序列；
- 本地发送状态和重试信息；
- 消息类型；
- 文本或附件 payload；
- 创建、发送、到达、已读时间；
- 撤回、引用、编辑等关系。

图片/视频还需要宽高、大小、格式、缩略图、原图地址、时长、校验值和本地缓存状态。字段存在的理由应与展示、传输、缓存和降级相连，而不是为了“字段齐全”。

# 8. Token、Session 与认证

Session 常由服务端保存会话状态，客户端持有会话标识；Token 更常把身份声明或随机凭证交给客户端携带，服务端可以有状态或无状态校验。

双 Token 常指短期 access token 和长期 refresh token：

- access token 过期后，用 refresh token 换新；
- 刷新需要串行化，避免多个请求同时刷新；
- 失败时清理凭证并回到登录；
- Token 应保存在受保护存储中，避免日志、截图和明文备份；
- 客户端混淆不能把硬编码密钥变成真正秘密。

# 9. Flutter、React Native、KMM 与 JNI

## Flutter

Flutter 通常自己完成布局和渲染。Widget 是不可变配置，Element 维护树中实例关系，RenderObject 负责布局与绘制，Layer 参与合成。

与原生通信可用 Platform Channel、FFI 或插件封装。需要关注线程、序列化成本、生命周期、异常和大数据传输。

## React Native

RN 面试常追问：

- JS 状态更新后如何触发组件协调与原生渲染；
- 组件生命周期；
- Bridge/JSI/TurboModule 等通信路径；
- 原生模块暴露和线程模型；
- 大量跨桥调用的性能成本。

## KMM/Compose Multiplatform

KMM 主要共享业务逻辑和数据层，平台 UI 是否共享取决于方案。必须说明哪些能力适合共享、哪些仍需平台实现，以及 expect/actual、线程、存储、网络和生命周期边界。

## JNI

JNI 连接 JVM 与 Native：

- JavaVM 通常代表进程级虚拟机；
- JNIEnv 与线程相关，不能跨线程直接复用；
- Native 方法绑定可以静态命名或动态注册；
- 需要管理局部/全局引用、异常、线程附着和数据拷贝。

Java/Native/Unity 等多层通信题的重点是定义清晰的接口、线程和所有权，不是只说“通过 JNI 调用”。

# 10. 动态化、混淆与包安全

- R8/ProGuard 可做压缩、优化和混淆，但不是密钥保护方案。
- 动态加载、插件化或 JSON 页面能提升发布灵活性，同时带来版本兼容、签名校验、资源隔离、崩溃回退和安全风险。
- 热更新必须定义允许修改的范围、审核、灰度、回滚和完整性校验。
- 反射、序列化和 JNI 入口需要正确 keep 规则，否则混淆后可能运行失败。

# 11. 可观测性

工程答案应包含观测：

- 请求 trace id 和耗时分段；
- DNS、连接、TLS、服务端和下载时间；
- 缓存命中率；
- 错误码、重试和取消原因；
- 数据版本和 UI 提交版本；
- 线上采样、隐私脱敏和日志容量。

没有观测的“优化”很难证明有效，也无法在偶现问题上建立因果链。
