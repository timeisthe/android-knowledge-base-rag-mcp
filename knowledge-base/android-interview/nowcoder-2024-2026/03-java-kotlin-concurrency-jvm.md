# 1. 并发问题的根：共享可变状态

线程安全问题不是“用了多线程就会出错”，而是多个执行流在没有正确同步关系时读写共享可变状态。

需要同时区分三类保证：

- 原子性：操作不会被其他线程观察到中间状态。
- 可见性：一个线程的写入能被另一个线程及时观察。
- 有序性：编译器和 CPU 的重排序不会破坏跨线程依赖。

只回答“加锁”不够。应先说明数据所有权、读写比例、竞争强度和允许的一致性级别，再选择机制。

# 2. volatile、synchronized、Lock 与 CAS

## volatile

volatile 主要提供可见性和特定的有序性约束。它不把复合操作自动变成原子操作，因此 i++ 即使作用于 volatile 变量也可能丢失更新。

适合：

- 状态标志；
- 单写多读；
- 配合双重检查锁发布已完整构造对象。

不适合：

- 计数累加；
- 需要跨多个字段维持不变量；
- 读改写复合逻辑。

## synchronized

synchronized 同时提供互斥、可见性和 happens-before 关系。锁对象不同，保护范围就不同：

- 实例同步方法锁当前对象；
- 静态同步方法锁 Class 对象；
- synchronized(obj) 锁指定对象。

同一线程可重入，因此同步方法内部调用同一把锁保护的方法不会自我死锁。静态同步方法和实例同步方法通常不是同一把锁。

关于“锁升级”，面试时不要把历史实现细节说成永恒规范。不同 JDK 版本可能移除或调整偏向锁等优化；核心应说明 JVM 会根据竞争情况选择不同的监视器实现和优化路径。

## ReentrantLock

相比 synchronized，Lock 能提供：

- 可中断获取；
- 超时尝试；
- 公平策略；
- 多个 Condition 条件队列；
- 更显式的加锁范围。

代价是必须在 finally 中释放，使用错误更容易造成泄漏锁。

## CAS 与 ABA

CAS 比较内存中的当前值和期望值，相等时更新。它通常依赖 CPU 原子指令，适合短小的无锁更新。

CAS 不代表没有竞争成本。高竞争下会自旋消耗 CPU；复杂不变量仍可能需要锁。ABA 问题是值从 A 变 B 又变 A，CAS 只比较值会误以为未变化，可使用版本戳或不可复用标识解决。

# 3. 线程池与任务编排

ThreadPoolExecutor 的关键参数：

- corePoolSize：核心工作线程目标数量。
- maximumPoolSize：队列满后允许扩展的最大线程数。
- keepAliveTime：非核心线程等空闲线程的回收时间。
- workQueue：等待执行的任务队列。
- threadFactory：线程命名、优先级和异常处理。
- rejectedExecutionHandler：饱和后的拒绝策略。

执行决策大致是：

未达到核心线程数 → 新建核心线程；
达到核心线程数 → 尝试入队；
队列满且未达最大线程数 → 新建额外线程；
线程和队列都饱和 → 执行拒绝策略。

队列类型会直接改变行为。无界队列会让 maximumPoolSize 基本失去作用，并把风险转成任务堆积和内存压力。

## 安全停止

不建议使用强制停止线程。合理方式是：

- 协作式取消标志；
- interrupt 中断阻塞；
- Future.cancel；
- ExecutorService.shutdown/shutdownNow；
- 任务在关键点检查取消并正确清理资源。

## 生产者消费者

核心不是背一段代码，而是维护以下不变量：

- 缓冲区容量有界；
- 空时消费者等待，满时生产者等待；
- 状态变化后唤醒正确的等待方；
- 条件检查必须用循环，防止虚假唤醒；
- 中断、关闭和异常要能传播。

可用 synchronized + wait/notifyAll、Lock + Condition，或直接使用 BlockingQueue。

# 4. HashMap 高频追问链

## 结构与 put

HashMap 使用数组桶保存节点。发生哈希冲突时，同一桶内使用链表；当桶内节点足够多且数组容量达到条件时，可树化为红黑树。

put 的核心步骤：

1. 对 key 的 hash 做扰动；
2. 根据数组长度定位桶；
3. 桶为空则插入；
4. 否则按 key 相等规则更新或追加；
5. 必要时树化；
6. size 超阈值后扩容。

扩容通常把容量翻倍。旧桶元素可依据 hash 的某一位分到原索引或原索引加旧容量，不必重新做完整取模。

## 为什么可变 key 危险

如果 key 的 hashCode 依赖可变字段，插入后修改字段，再 get 时可能定位到不同桶，导致逻辑上“同一个对象”取不到值。解决方法不是事后重写 equals/hashCode，而是让作为 key 的身份字段不可变，并保证 equals 与 hashCode 一致。

## 为什么线程不安全

并发 put、resize 或结构修改会产生丢失更新、数据不可见、遍历异常和结构不一致。不能只回答“可能死循环”，因为那与具体 JDK 实现版本有关。

## ConcurrentHashMap

现代实现主要通过 CAS、桶级同步和 volatile 可见性控制，避免整张表一把大锁。它仍不能自动保证跨多个操作的业务原子性；“先判断再 put”应使用 compute、putIfAbsent 等原子复合 API，或在更高层加锁。

## ArrayList 与 LinkedList

- ArrayList 随机访问快，因为连续数组可按索引寻址。
- LinkedList 已知节点位置时插入/删除是 O(1)，但按索引找到位置仍是 O(n)。
- 实际工程中 LinkedList 还存在额外对象、指针和缓存局部性较差的问题，不能只比较理论复杂度。

# 5. JVM、对象创建、类加载与 GC

## new 对象的逻辑过程

典型过程包括：

- 类是否已加载和初始化；
- 分配对象内存；
- 清零并设置对象头；
- 执行实例字段初始化、实例代码块和构造函数。

具体分配可能使用指针碰撞、空闲列表和线程本地分配缓冲区，取决于堆布局和 JVM 实现。

## 类加载

常见阶段是加载、链接、初始化。链接包括验证、准备和解析。

初始化阶段执行类初始化方法，主要来自静态字段赋值和静态代码块。静态成员访问、new、反射等可能触发初始化，但常量内联和仅访问父类静态字段等场景有边界。

## GC

常见算法思想：

- 标记清除：简单，但产生碎片。
- 标记整理：移动存活对象，减少碎片。
- 复制：把存活对象复制到另一空间，适合存活率低的区域。
- 分代：依据对象生命周期特征组合策略。

Android 面试中的重点往往不是背收集器名称，而是能解释：

- 为什么 Java 有 GC 仍会泄漏；
- 哪些引用链阻止回收；
- 如何从堆快照定位；
- 优化是否只是把 OOM 推迟。

# 6. Kotlin、协程与 Flow

## Kotlin 相比 Java 的工程优势

常见优势包括空安全、扩展函数、数据类、密封类型、函数式 API、协程和更紧凑的语法。正确答案必须同时承认成本：编译产物、滥用高阶函数的分配、互操作边界和团队可读性。

## by lazy

lazy 把值的创建推迟到第一次访问，并缓存结果。不同线程安全模式有不同同步语义。面试中可类比 Java 的延迟初始化，但要说明“是否线程安全、初始化失败是否重试、生命周期由谁持有”。

## 协程与线程

协程是可挂起任务，需要在线程上执行。它通过状态机保存挂起点附近的局部状态，在异步结果就绪后由调度器恢复，不必阻塞原线程。

“挂起不消耗 CPU”不是因为代码神奇暂停，而是挂起函数把未完成工作注册为回调/continuation，当前线程可以去执行别的任务；恢复时再从状态机相应位置继续。

## 结构化并发

协程应属于明确的 CoroutineScope。父任务负责子任务的生命周期，取消和异常能沿结构传播。

在 Android 中：

- lifecycleScope 随 Lifecycle 销毁取消；
- viewModelScope 随 ViewModel clear 取消；
- repeatOnLifecycle 适合在可见状态收集 Flow，并在状态离开时取消子收集。

不要把 GlobalScope 当成解决生命周期问题的方式。

## StateFlow 与 UI

StateFlow 表示一个可观察的当前状态。Compose 收集后，状态值变化会使读取该状态的组合范围失效并重新组合。

需要注意：

- 新值与旧值相等时可能不会产生预期更新；
- 状态对象应尽量不可变；
- 一次性事件不应简单塞进可重复读取的页面状态；
- Flow 取消点和异常处理要与页面生命周期一致。

# 7. 高频误区

- volatile 不能保证所有线程安全问题。
- synchronized 锁的是对象/类，不是“代码文字”。
- ConcurrentHashMap 不能保证多个 API 组合的原子性。
- 协程不是不需要线程，也不是自动并行。
- 双重检查锁缺少 volatile 可能发布未完整初始化对象。
- 两个对象互相引用并不自动构成内存泄漏。
- LinkedList 插入不总是 O(1)，还要计入定位节点的成本。
