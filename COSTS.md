# 成本估算与优化建议

当前默认使用本地 `Qwen3-Embedding-0.6B`，不会产生按 token 计费；成本主要是首次下载、磁盘空间和本机计算时间。以下 OpenAI 部分仅用于切换到 `text-embedding-3-small` 时参考，实际价格请以 [OpenAI API Pricing](https://openai.com/api/pricing/) 为准。

## 基础估算

公式：

```text
费用 = 输入 token 数 ÷ 1,000,000 × 每百万 token 单价
```

| 场景 | Token 数 | 估算费用 |
|---|---:|---:|
| 初始化 100 篇，每篇 3,000 tokens | 300,000 | $0.006 |
| 每月更新 40 篇，每篇 3,000 tokens | 120,000 | $0.0024 |
| 每月 5,000 次查询，每次 50 tokens | 250,000 | $0.005 |
| 初始化 500 篇，每篇 3,000 tokens | 1,500,000 | $0.03 |

原始目标中的“每月更新 40 篇约 $0.012”与上述单价、篇幅不一致；按 `40 × 3,000 × $0.02 / 1M`，正确结果是约 **$0.0024**。若每月总计处理 600,000 tokens，费用才约 $0.012。

ChromaDB 和 Qwen3 均在本机运行，无按查询计费。使用 OpenAI Provider 时，才会产生嵌入输入费用。

## 当前实现的成本控制

- 每篇文章一个向量单元，frontmatter 不参与嵌入。
- 查询只返回摘要；完整正文由 `get_document` 按需获取。
- create/update 只重算目标文档，不扫描全库。
- OpenAI SDK 一次批量提交多个文档，减少网络往返。
- `OPENAI_EMBEDDING_DIMENSIONS` 可在 v3 模型上缩短向量，降低 Chroma 存储和搜索内存。

## 规模增长后的优化

### 1. 增加内容哈希

在元数据中存储正文哈希，`reindex_all` 只重算发生变化的文件。当前规模下全量重建费用很低，先保持实现简单可靠。

### 2. 超长文档改为分节索引

当前严格按“一篇文章一个向量”实现。若单篇增长到 8,000～12,000 tokens 且主题跨度很大，可按二级标题建立子向量，同时保留父文档路径；搜索返回命中章节，再由 `get_document` 读取全文。

### 3. 选择合适维度

先用模型默认维度建立质量基线，再评估 512、768 或 1024 等缩短维度。更改维度后必须使用新 Collection 或执行全量重建。

### 4. 缓存重复查询

对规范化后的高频问题缓存查询向量。只有查询量明显增长后再引入，避免缓存失效复杂度超过节省费用。

### 5. 控制 top_k

默认 5 篇适合编辑任务。事实明确时可取 2～3 篇；跨主题综述可取 5～8 篇。top_k 影响的是 AI 上下文，而非 Chroma 向量化费用。
